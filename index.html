单例模式

某些类在只需要一个实例的时候，就可以用到单例模式。这个实例由类来保存。

注意点：

  1. 有且只有一个实例。
  2. 这个实例由类自己创建。
  3. 可以让外部使用到这个类。

所以一般：

  1. 将唯一的实例作为静态成员。
  2. 将构造函数私有化，禁止外部调用构造函数。

应用场景

  1. 频繁创建和销毁的对象
  2. 大对象
  3. 工具类
  4. 频繁访问数据库或文件的对象

六种关系

  1. 依赖关系
  2. 泛化关系
  3. 实现关系
  4. 关联关系
  5. 聚合关系
  6. 组合关系

依赖关系

虚线箭头 ..>

依赖关系是一种通用的说法，其他类型的关系都是依赖关系的特例。

其他类直接使用或作为泛型：

  • 出现在域中
  • 出现在方法类型中
  • 出现在方法中
C ..> W
C ..> X
C ..> Y
C ..> Z

泛化关系

实线三角箭头 --|>

子类 继承 父类。

D --|> B

实现关系

虚线三角箭头 ..|>

子类 实现 接口。

C ..|> I

关联关系

实线箭头 -->

具有 导航性 。

单向一对一

A --> B

双向一对一

A <--> B

聚合关系

实线菱形空心箭头 --<>

表示整体和部分的关系，整体与部分 可以 分开。

具有 导航性 和 多重性 。

C <>-- X
C <>-- Y

组合关系

实线菱形实心箭头 --<*>

表示整体和部分的关系，整体与部分 不能 分开。

具有 导航性 和 多重性 。

C <*>-- X
C <*>-- Y

设计模式的七条原则

发表于 2020-09-16

七条原则

  1. 开闭原则
  2. 单一职责原则
  3. 接口隔离原则
  4. 依赖倒转（倒置）原则
  5. 里氏替换（代换）原则
  6. 迪米特法则（最少知道原则）
  7. 合成复用原则

开闭原则

对拓展开放，对修改关闭。

最重要的一条，其他条目都是对它的解释。

添加新功能时，写新类，而不对已经写完的类进行修改。


单一职责原则

一个类只负责一个功能领域中的相应职责。

在一个类分则多项职责时对其进行拆分。


接口隔离原则

客户端不应该依赖它不需要的接口。

类间的依赖关系应该建立在最小的接口上。

类不需要实现接口中所有方法时对接口进行拆分，拆分成多个接口后类选择需要的方法进行实现。


依赖倒转原则

抽象不应该依赖于细节，细节应该依赖于抽象。

高层模块不应该依赖于低层模块，两者都应该依赖于自身的抽象。

面向接口编程。


里氏替换原则

任何基类可以出现的地方，子类一定可以出现。

子类尽量不重写基类方法。


迪米特法则

一个对象应该对其他对象有最少的了解。

只与朋友类交流，方法内部不要出现非朋友类。

朋友类：出现在成员变量与方法类型中的类。


合成复用原则

尽量使用对象组合/聚合, 而不是继承关系达到软件复用的目的。

需要用到另一个类时，自身不去将其继承，而将其作为成员变量，通过 set 方法（组合）或 在构造函数中或直接 new 进行构造（聚合）。

Java 的泛型是假的泛型

有个东西叫类型擦除。

这些东西是一样的。 JVM 中（也就是 Java 运行期间）是没有泛型的，全是 Object ！

所以 Java 的泛型在本质上其实就是 Object 与其子类 之间的强制类型转换。

这样也就能解释得通上面为什么一会儿报错一会儿不报错了。

所以也有人说： Java 的泛型是假的泛型。

创建博客

  1. Windows环境用 gitbash 来进行操作会方便很多

  2. 环境搭建

    1. nodejs 与 npm（nodejs.org）
      • 安装nodejs与npm：sudo apt install nodejs
      • 检查nodejs：node -v
      • 检查npm：npm -v
    2. cnpm（cnpm使用国内路线，可选安装）
      • 安装：npm install -g cnpm --registry=https://registry.npm.taobao.org
      • 检查：npm -v
    3. hexo
      • 安装：npm install -g hexo-cli
      • 检查：hexo -v

  3. 创建并进入博客文件路径

  4. 初始化博客：sudo hexo init

  5. 在github中创建仓库

    • 仓库名：username.github.io

  6. deployer

    • 安装：npm install --save hexo-deployer-git

  7. 修改配置文件 _config.yml

    将文件底部 deploy 部分修改为

     deploy:
     type: git
     repo: https://github.com/username/username.github.io.git
     branch: master

  8. 推送到仓库

基本操作

在博客文件夹下进行以下命令行操作

清除配置

hexo clean 或 hexo cl

重新生成

hexo generate 或 hexo g

本地挂载

hexo server 或 hexo s

推送到仓库

hexo deploy 或 hexo d

浏览器打开 localhost:4000 进行预览

换皮肤

  1. 将皮肤的仓库克隆至目录：Blog/themes/themename

  2. 修改配置文件 _config.yml

     theme: themename

  3. 清除配置

  4. 重新生成

  5. 推送到仓库
