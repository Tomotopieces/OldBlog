设计模式-访问者模式与观察者模式

发表于 2020-09-24 更新于 2020-10-22

访问者模式

违反了 迪米特法则 和 依赖倒转原则 的奇怪模式，目的是将数据结构与数据操作分离，降低耦合性，但没有将访问主体与访问者解耦。这部分的解耦交给其他的设计模式。

访问者只需要关心我访问了什么，这给我来带了什么样的影响。被访问的只需要关心是谁访问了我，给我带来了什么影响。他们之间是如何产生这样访问的，并不需要关心。

by chaos

应用场景

适用于数据结构相对稳定算法又易变化的系统，若系统数据结构对象易于变化，则不适合使用访问者模式。

类图

Visitor Pattern:

Visitor    Client    ObjectStructure             Element
   |         |              |                       |
   | -----<> | <>---------- | <*>------------------ |
   |         |              |                       |
   |       Client    ObjectStructure                |
   |                                                |
   |       ConcreteVisitor    ConcreteElement       |
   |              |                  |              |
   | <|.......... |                  | ..........|> |
   |              |                  |              |
Visitor    ConcreteVisitor    ConcreteElement    Element

---
ObjectStructure
- elements: Element
+ accept(Visitor)
---
<<interface>>
Element
+ accept(Visitor)
---
<<interface>>
Visitor
+ visit(ConcreteElement)

观察者模式

感觉上它的另一个名字 发布-订阅模式 更为恰当，因为数据是被观察主体主动提供的。

一个发布者 Observable 对多个订阅者 Observer 。

也违反了 迪米特法则 。（迪米特：？）

应用场景

  1. 当一个抽象的多个实现依赖另一个抽象的实例，需要得到更新通知。
  2. 可设计成链式触发机制。

类图

Observable    Observer    ConcreteObserver
    |            |               |
    | <>-----<*> | <|........... |
    |            |               |
    |         Observer    ConcreteObserver
    |
    |         Subject
    |            |
    | <|-------- |
    |            |
Observable    Subject

---
<<abstract>>
Observable
- List<Observer>
+ addObserver(Observer)
+ deleteObserver(Observer)
+ notifyObservers()
---
<<interface>>
Observer
+ update(Observable)

区别

  1. 访问者模式可能会对主体进行修改，观察者模式一般不会。
  2. 访问者模式主动获取主体信息，观察者模式被动接受主体信息。

设计模式-建造者模式

发表于 2020-09-18 更新于 2020-10-22

建造者模式

核心角色：

  1. 产品
  2. 抽象建造者
  3. 具体建造者
  4. 指挥者

应用场景

当一个类的构建过程本身就是一个复杂的流程时，就有必要将构建过程单独提出来做成一个类或模块。这个类或模块就是建造者。


类图

Builder Pattern:

Builder    Product    Director    Client
   |          |          |           |
   | <*>----- | <....... |           |
   |          |          |           |
   |       Product       |           |
   |                     |           |
   | ----------------<*> |           |
   |                     |           |
   |                  Director       |
   |                                 |
   |         ConcreteBuilder         |
   |                |                |
   | <|------------ |                |
   |                |                |
Builder      ConcreteBuilder      Client

Client 根据实际情况会与其他各个类有不同的关系。


Director ?

如果建造过程简单，完全可以将 Director 合并到 Builder 中。

Builder 中的方法负责实现建造过程中的每个细节，而 Director 负责将所有的细节按合理的逻辑顺序组合在一起。可以算是为了迎合 单一职责原则 而存在的东西。本质上创建 Builder 类也是将构建这一职责从原本的类中拆分出来从而满足单一职责原则的做法。

UML类图中的六种关系

发表于 2020-09-16 更新于 2020-10-22

六种关系

  1. 依赖关系
  2. 泛化关系
  3. 实现关系
  4. 关联关系
  5. 聚合关系
  6. 组合关系

依赖关系

虚线箭头 ..>

依赖关系是一种通用的说法，其他类型的关系都是依赖关系的特例。

其他类直接使用或作为泛型：

  • 出现在域中
  • 出现在方法类型中
  • 出现在方法中
C ..> W
C ..> X
C ..> Y
C ..> Z

泛化关系

实线三角箭头 --|>

子类 继承 父类。

D --|> B

实现关系

虚线三角箭头 ..|>

子类 实现 接口。

C ..|> I

关联关系

实线箭头 -->

具有 导航性 。

单向一对一

A --> B

双向一对一

A <--> B

聚合关系

实线菱形空心箭头 --<>

表示整体和部分的关系，整体与部分 可以 分开。

具有 导航性 和 多重性 。

C <>-- X
C <>-- Y

组合关系

实线菱形实心箭头 --<*>

表示整体和部分的关系，整体与部分 不能 分开。

具有 导航性 和 多重性 。

C <*>-- X
C <*>-- Y

设计模式的七条原则

发表于 2020-09-16 更新于 2020-10-22

七条原则

  1. 开闭原则
  2. 单一职责原则
  3. 接口隔离原则
  4. 依赖倒转（倒置）原则
  5. 里氏替换（代换）原则
  6. 迪米特法则（最少知道原则）
  7. 合成复用原则

开闭原则

对拓展开放，对修改关闭。

最重要的一条，其他条目都是对它的解释。

添加新功能时，写新类，而不对已经写完的类进行修改。


单一职责原则

一个类只负责一个功能领域中的相应职责。

在一个类分则多项职责时对其进行拆分。


接口隔离原则

客户端不应该依赖它不需要的接口。

类间的依赖关系应该建立在最小的接口上。

类不需要实现接口中所有方法时对接口进行拆分，拆分成多个接口后类选择需要的方法进行实现。


依赖倒转原则

抽象不应该依赖于细节，细节应该依赖于抽象。

高层模块不应该依赖于低层模块，两者都应该依赖于自身的抽象。

面向接口编程。


里氏替换原则

任何基类可以出现的地方，子类一定可以出现。

子类尽量不重写基类方法。


迪米特法则

一个对象应该对其他对象有最少的了解。

只与朋友类交流，方法内部不要出现非朋友类。

朋友类：出现在成员变量与方法类型中的类。


合成复用原则

尽量使用对象组合/聚合, 而不是继承关系达到软件复用的目的。

需要用到另一个类时，自身不去将其继承，而将其作为成员变量，通过 set 方法（组合）或 在构造函数中或直接 new 进行构造（聚合）。
