OpenGL-Triangle

发表于 2020-09-27

三角形

众所周知，三个点可以形成一个面，且 OpenGL 的图形绘制就是按各种三角形来的。所以能画一个三角形，就能画所有的东西。

VAO / VBO / EBO / Shader / Program

VAO: Vertex Array Object

顶点数组对象

用来绑定同一个图形的 VBO ， EBO 和 Program 。在绑定后，图形所有的属性调用都存在这里面，方便绘制。可以理解为一个简单的封装。

使用前记得绑定，使用后记得解绑。

VBO: Vertex Buffer Object

顶点缓冲对象

用来存一个数组，这个数组包含了一个图形的所有顶点。

假设有一三角形，顶点坐标可以为：

    vertex = 
    [
           0,  0.5, 0,
        -0.5, -0.5, 0,
         0.5, -0.5, 0
    ]

因为是 2D 的三角形，所以第三个 z 轴值忽略。

坐标值都在 -1.0 ~ 1.0 之间，叫标准化设备坐标。

EBO: Element Buffer Object

索引缓冲对象

用来存一个数组，这个数组描述了如何将图形划分为若干个三角形。

假设有一举行，顶点数组为：

    vertex = 
    [
        -0.5,  0.5, 0,
         0.5,  0.5, 0,
        -0.5, -0.5, 0,
         0.5, -0.5, 0
    ]

OpenGL 要求用户告知如何 将这个矩形拆成若干个三角形 ，再进行绘制。所以才有了 EBO 这东西，相当于给告诉 OpenGL ，请按以下顶点顺序来绘制各个三角形：

    indices =
    [
        0, 1, 2,
        2, 3, 0
    ]

这样就将矩形以左上到右下的对角线来拆分成了两个三角形。
所以 EBO 也被叫做 IBO: Indices Buffer Object 。

Shader

着色器

用来表述如何绘制。

可吞吐少量数据。

GPU 上全是这些小 Shader 。

Program

着色器程序

用来链接不同的着色器。

进行绘制时，至少需要一个 VertexShader 顶点着色器 和一个 FragmentShader 片段着色器。 Program 负责将两者链接起来。

使用时的一般流程。

在代码中使用 VAO / VBO / EBO 时，一般使用 glGenXXX 或者 glCreateXXX 函数来创建一个对象，用户用一个 unsigned int 来存放函数返回的对象 id 。在修改或调用时，也一般通过调用 glGetXXX ， glSetXXX 或者 glBindXXX 函数，传入 id 来使用。

VAO 需要绑定为 VertexArray ， VBO 和 EBO 需要在对应的 Buffer 内写入数据。

Shader 在声明创建后需要从源获取，然后编译。

Program 在声明创建后需要将 Shader 附加上来，然后链接。

OpenGL-环境搭建

发表于 2020-09-26

下载必要库

  • GLFW
    • 针对 OpenGL 的 C 语言库。
    • 下载：https://www.glfw.org/download.html
    • 下载 32 位版本，因为 64 位版本不稳定。
  • GLEW
    • 用于代替 GLAD 来管理函数指针。
    • 下载：https://sourceforge.net/projects/glew/

将项目与库连接

  1. 将 GLFW 和 GLEW 解压到一个新文件夹中，文件夹名建议为 OpenGL。目录结构如下。

     - OpenGL
   - glfw-x.x.x.bin.WIN32
   - glew-x.x.x

  2. 使用 Visual Studio 创建项目，创建一个 .cpp 格式的文件，比如 main.cpp 。

  3. 设置项目属性

    1. 配置：所有配置
    2. 平台：所有平台
    3. C/C++ -> 常规 -> 附加包含内容 -> 编辑 -> 新行
      1. \OpenGL\glfw-x.x.x.bin.WIN32\include
      2. \OpenGL\glew-x.x.x\include
    4. 链接器 -> 常规 -> 附加库目录 -> 编辑 -> 新行
      1. \OpenGL\glfw-x.x.x.bin.WIN32\lib-vc20xx
      2. \OpenGL\glew-x.x.x\lib\Release\Win32
    5. 链接器 -> 输入 -> 附加依赖项 -> 编辑
      1. opengl32.lib
      2. glfw3.lib
      3. glew32s.lib

设计模式-访问者模式与观察者模式

发表于 2020-09-24

访问者模式

违反了 迪米特法则 和 依赖倒转原则 的奇怪模式，目的是将数据结构与数据操作分离，降低耦合性，但没有将访问主体与访问者解耦。这部分的解耦交给其他的设计模式。

访问者只需要关心我访问了什么，这给我来带了什么样的影响。被访问的只需要关心是谁访问了我，给我带来了什么影响。他们之间是如何产生这样访问的，并不需要关心。

by chaos

应用场景

适用于数据结构相对稳定算法又易变化的系统，若系统数据结构对象易于变化，则不适合使用访问者模式。

类图

Visitor Pattern:

Visitor    Client    ObjectStructure             Element
   |         |              |                       |
   | -----<> | <>---------- | <*>------------------ |
   |         |              |                       |
   |       Client    ObjectStructure                |
   |                                                |
   |       ConcreteVisitor    ConcreteElement       |
   |              |                  |              |
   | <|.......... |                  | ..........|> |
   |              |                  |              |
Visitor    ConcreteVisitor    ConcreteElement    Element

---
ObjectStructure
- elements: Element
+ accept(Visitor)
---
<<interface>>
Element
+ accept(Visitor)
---
<<interface>>
Visitor
+ visit(ConcreteElement)

观察者模式

感觉上它的另一个名字 发布-订阅模式 更为恰当，因为数据是被观察主体主动提供的。

一个发布者 Observable 对多个订阅者 Observer 。

也违反了 迪米特法则 。（迪米特：？）

应用场景

  1. 当一个抽象的多个实现依赖另一个抽象的实例，需要得到更新通知。
  2. 可设计成链式触发机制。

类图

Observable    Observer    ConcreteObserver
    |            |               |
    | <>-----<*> | <|........... |
    |            |               |
    |         Observer    ConcreteObserver
    |
    |         Subject
    |            |
    | <|-------- |
    |            |
Observable    Subject

---
<<abstract>>
Observable
- List<Observer>
+ addObserver(Observer)
+ deleteObserver(Observer)
+ notifyObservers()
---
<<interface>>
Observer
+ update(Observable)

区别

  1. 访问者模式可能会对主体进行修改，观察者模式一般不会。
  2. 访问者模式主动获取主体信息，观察者模式被动接受主体信息。

设计模式-桥接模式与装饰器模式

发表于 2020-09-22

两者都用于处理类数量爆炸的情况。

桥接模式

将抽象类作为具体实现类与功能接口的桥。

应用场景

当类需要在多个方面进行拓展时，使用桥接模式。

类图

Bridged Pattern:

Product       Dimension       ConcreteDimension
   |              |                   |
   | <>---------- | <|............... |
   |              |                   |
   |          Dimension               |
   |                                  |
   |       ConcretePruduct            |
   |              |                   |
   | <|---------- | <>--------------- |
   |              |                   |
Product    ConcretePruduct    ConcreteDimension

ConcreteProduct 与 ConcreteDimension 根据具体情况，可能是组合或者聚合关系。


装饰器模式

众所周知，青蛙不会飞。但是我们可以用 FrogFlyDecorator 让青蛙飞起来。

应用场景

为产品添加新的功能时，使用装饰器模式。

在装饰器中为产品添加新功能。

类图

Decorator Pattern:

Product    Decorator    ConcreteDecorator
   |           |                |
   | <|----<*> | <|------------ |
   |           |                |
   |       Decoretor    ConcreteDecorator
   |
   |       ConcretePruduct
   |              |
   | <|---------- |
   |              |
Product    ConcretePruduct

Decorator 泛化并组合了一个抽象的 Product ， ConcreteDecorator 为 Product 添加新功能或修改原有功能。

泛化的目的是为了能嵌套，做到类似这样的效果：

这是一只能说话能跑步能飞的青蛙。

手抓饼加料的过程可以同理。


二者区别

  1. 桥接模式将相关的事物组合或聚合在一起；装饰器模式为已经存在的事物添加新的功能。
  2. 桥接模式中的产品实现类与接口实现类继承自不同的类/接口，没有主次之分；装饰器模式的产品实现类与装饰器类都继承自抽象产品类，产品为主，装饰器为次。

设计模式-建造者模式

发表于 2020-09-18 更新于 2020-09-21

建造者模式

核心角色：

  1. 产品
  2. 抽象建造者
  3. 具体建造者
  4. 指挥者

应用场景

当一个类的构建过程本身就是一个复杂的流程时，就有必要将构建过程单独提出来做成一个类或模块。这个类或模块就是建造者。


类图

Builder Pattern:

Builder    Product    Director    Client
   |          |          |           |
   | <*>----- | <....... |           |
   |          |          |           |
   |       Product       |           |
   |                     |           |
   | ----------------<*> |           |
   |                     |           |
   |                  Director       |
   |                                 |
   |         ConcreteBuilder         |
   |                |                |
   | <|------------ |                |
   |                |                |
Builder      ConcreteBuilder      Client

Client 根据实际情况会与其他各个类有不同的关系。


Director ?

如果建造过程简单，完全可以将 Director 合并到 Builder 中。

Builder 中的方法负责实现建造过程中的每个细节，而 Director 负责将所有的细节按合理的逻辑顺序组合在一起。可以算是为了迎合 单一职责原则 而存在的东西。本质上创建 Builder 类也是将构建这一职责从原本的类中拆分出来从而满足单一职责原则的做法。
